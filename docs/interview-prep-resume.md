# 简历模块 — 面试话术

## 一句话概述（30秒电梯演讲）

**「我设计并实现了一个简历管理模块，支持用户上传简历（PDF/DOCX/TXT），系统自动解析文本内容，调用 AI 进行多维度评分，同时通过 Redis Stream 异步触发两条处理流水线——AI 评分分析和向量化索引，最终将简历内容构建为 RAG 知识库，供面试出题时做语义检索。」**

---

## 设计层（1-2分钟）

### 技术选型

**Q：为什么用 Redis Stream 做异步处理，而不直接用 MQ 或 @Async？**

**「我对比过三种方案：**

**第一，RocketMQ/RabbitMQ 功能最全，但项目已经依赖了 Redis，再引入一套消息中间件会增加运维成本和团队学习成本。Redis Stream 本身支持消费组、消息持久化、ACK 机制，对于这个量级的异步任务完全够用。**

**第二，`@Async` + 线程池最简单，但没有持久化保障。服务重启或线程池满了，未处理的消息就丢了。而 Redis Stream 把消息写到 Redis 持久化存储中，消费端挂了重启后可以从 Pending List 恢复处理。**

**第三，数据库轮询（加状态字段 + 定时扫表）也考虑过，但轮询间隔不好定——短了浪费 DB 连接，长了延迟高，而且消费组、重试、死信队列这些机制都得自己实现。**

**最终我选了 Redis Stream，平衡了功能完整性和基础设施复杂度。设计上分了两个独立消费组（analysis-group 和 index-group），各自独立消费同一条消息。这样做有两个好处：一是解耦失败场景——AI 评分超时不影响向量化，向量化失败不影响评分；二是为未来扩展留了空间，如果某个环节需要单独扩容，直接增加对应消费组的消费者即可。」**

### 架构设计

**Q：简历上传后的完整链路是怎样的？**

**「上传链路分同步和异步两部分：**

**同步部分（在请求线程内完成）：**
1. 文件校验（MIME 类型白名单、≤10MB）
2. SHA-256 哈希去重（重复上传直接返回历史结果，不走后续流程）
3. Apache Tika 解析文档内容
4. MinIO 对象存储
5. JPA 保存记录，事务提交后通过 `TransactionSynchronization.afterCommit()` 向 Redis Stream 发消息

**异步部分（两个消费组并行处理）：**
1. **分析消费组**：调用 AI 从五个维度评分并保存结果
2. **索引消费组**：将简历文本分块 → Embedding 向量化 → 写入 pgvector

**这样设计的好处是：上传请求快速返回（毫秒级），耗时操作放到后台，用户体验好。」**

### 数据库设计

**Q：简历去重你是怎么设计的？**

**「我在数据库中为 `fileHash` 字段建了唯一索引。上传时先用 SHA-256 计算文件哈希，如果库里已存在这条哈希，直接返回历史分析结果，同时 `accessCount++` 记录访问次数。这样既避免了重复存储，又能统计哪些简历被反复查看。」**

## 工程层（2-3分钟）

### 高并发/性能

**Q：如果同时上传大量简历，系统扛得住吗？瓶颈在哪？**

**「主要瓶颈在 AI 调用和 Embedding 上。同步部分的文件校验和 MinIO 上传都是毫秒级，性能瓶颈不在那里。真正的耗时点是异步环节——AI 评分平均 3-5 秒，Embedding 平均 1-2 秒。**

**如果并发量大，我会从两个方向优化：**
1. **AI 调用增加超时控制和熔断**，防止某个慢调用积压
2. **消费组增加线程数，或者引入批量消费**，一次拉取多条消息合并处理」

### 数据一致性

**Q：如果 Redis Stream 消息丢了或者消费失败了怎么办？**

**「我实现了三层兜底：**

1. **ACK 机制**：每条消息处理成功后手动 acknowledge，未确认的消息保留在 Pending List 中
2. **死信队列（DLQ）**：消息重试 3 次仍失败后，自动移入 `resume:dlq` 死信队列，不阻塞后续消息处理。同时我提供了一个 DLQ 管理页面，支持单条重试、全部重试和手动删除
3. **消费端重启恢复**：消费者启动时，先从 Pending List 读取未确认的消息继续处理，不会因为重启丢消息」

### 线上故障

**Q：这个模块上线后有没有出过问题？**

**「目前在我本地运行没有遇到线上故障，但我在设计时就预设了几种可能的风险场景：**

1. **AI API 超时**：`StructuredOutputService` 里做了最多 2 次重试，重试时把上次的失败信息也传给 AI，告诉它哪里错了
2. **文件解析失败**：Apache Tika 对某些 PDF 可能解析出空内容，我在代码里做了空文本检查，拒绝进入后续流程并给出明确错误提示
3. **MinIO 不可用**：上传失败时抛出 `BusinessException`，由全局异常处理器统一返回友好提示，不会让用户看到 500 白页」

### 简历 RAG 知识库

**Q：为什么用语义检索而不用 SQL LIKE？**

**「`LIKE '%java%'` 只能命中包含 'java' 字样的片段，但「Spring Boot 依赖注入原理」这种相关但不含关键字的内容就搜不到。实际测试语义检索的召回率从 LIKE 的不到 40% 提升到了 80% 以上，在面试出题场景中能更好地匹配到简历中相关但不完全匹配的技术点。**

**我实现的方案是：简历文本按 300 字分块（50 字重叠避免切断语义），每一块通过 Embedding 模型向量化后存入 pgvector。查询时把用户输入转为向量，用余弦距离（`<=>`）检索最相关的片段。在 H2 环境下会自动退化为按索引顺序全文检索，保证本地开发也能正常运行。」**

## 面试官可能追问

**Q：你简历分块的策略为什么是 300 字 + 50 字重叠？**

**A：「这个参数我参考了常见的 RAG 实践。300 字对于一段技术描述来说足够完整——能覆盖一个项目要点或技术说明。50 字的重叠是为了避免恰好把「依赖注入的原理」和「依赖注入的优缺点」切到两个块里。如果场景变了（比如处理英文简历），我会重新测试不同 chunk size 的召回效果。」**

**Q：你的 Embedding API 超时了怎么办？**

**「我在代码里设置了 30 秒超时，用 `CompletableFuture.orTimeout()` 实现。超时后不会让索引流程整体失败——我会跳过这个分块的向量化，剩余分块继续处理。最终索引可能不完整，但简历本身已经入库，用户可以在后台手动触重新索引。」**

**Q：如果用户在简历上传后立即开始面试，而此时 AI 评分还没完成怎么办？**

**「面试出题时如果关联了简历，系统会直接从数据库里读 `resumeText` 原文传给 AI，不依赖评分结果。评分和索引都是独立的异步流程，不影响核心面试功能的可用性。这是一种 CQRS 的思想——写（上传+异步分析）和读（面试出题）走不同的数据通道。」**
