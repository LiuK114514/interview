# 面试模块 — 面试话术

## 一句话概述（30秒电梯演讲）

**「我实现了一个完整的 AI 模拟面试引擎，支持从简历/技能方向智能出题、逐题 AI 实时评分、以及综合分析报告生成。核心难点在于让大模型稳定输出结构化的题目和评分结果——我设计了一套包含重试和本地 JSON 修复的结构化输出框架，保证了 AI 输出的可靠解析。」**

---

## 设计层（1-2分钟）

### 技术选型

**Q：为什么没有直接用 Spring AI 的 ChatClient 调用，而是自己封装了一层 StructuredOutputService？**

**「我设计 `StructuredOutputService` 主要解决两个问题：**

**第一，复用。项目中需要结构化输出的地方不止一个——面试出题、答案评估、简历评分、报告生成都用到了。如果每个地方都重复写 SystemPrompt + Converter + 重试逻辑，代码会非常冗余。所以我抽出了一个通用方法 `invoke(chatClient, systemPrompt, userPrompt, converter, taskName)`，调用方只需要传提示词和转换器，像这样：**

```java
InterviewQuestions questions = structuredOutputService.invoke(
    chatClient, systemPrompt, "请开始出题", converter, "面试出题");
```

**第二，可靠性。Spring AI 的 `BeanOutputConverter` 本身只做 JSON 到对象的转换，不处理 AI 输出格式异常的情况。我在这层加了两道防线：本地 JSON 引号修复和带错误上下文的重试机制。」**

**Q：为什么用 JSON 存题目和报告，而不是建关联表？**

**「面试题目的结构比较简单——就是 5 道题的列表，字段固定（index、question、category）。如果建一张 `questions` 表加外键关联，需要多一次 JOIN，而且题目在生成后不会被单独查询或修改。用 JSON 存在 `questionsJson` 字段里，一次查询就能拿到全部数据，反序列化成本也很低。**

**这是一个权衡：如果将来需要单独查询题目的答案率、正确率等统计数据，我会把这个 JSON 拆成独立的表。但在当前阶段，JSON 的方式更简单高效。」**

### 架构设计

**Q：面试流程的状态机是什么样的？**

**「面试会话有四个状态：**

```
CREATED → IN_PROGRESS → COMPLETED → EVALUATED
```

- **CREATED**：AI 已出题，会话刚创建
- **IN_PROGRESS**：用户已提交至少一道答案
- **COMPLETED**：全部题目答完
- **EVALUATED**：综合报告已生成

`nextQuestion()` 接口会检查哪些题已经答过（通过查询 `interview_answers` 表），跳过已答题返回下一道未答题。全部答完时自动将状态改为 COMPLETED。`generateReport()` 只允许在 COMPLETED 状态下调用，生成后更新为 EVALUATED 并缓存报告结果，后续重复请求直接返回缓存。」**

### 事务边界控制

**Q：我注意到你的 submitAnswer 方法没有加 @Transactional，而其他方法加了，为什么？**

**「这是刻意的设计。AI 调用平均耗时 3-5 秒，如果放在事务内，数据库连接会被长时间占用。假设 HikariCP 默认池大小是 10，10 个请求进来都在等 AI 返回，连接池就耗尽了，其他需要事务的操作（比如上传简历、查询列表）就会阻塞或超时。**

**还有一个更隐蔽的问题：长事务会导致数据库 MVCC 的旧版本数据无法清理，影响查询性能。**

**所以我的做法是：`submitAnswer` 中不开启事务，在无事务上下文中调用 AI。AI 返回结果后，由 JPA 的 `save()` 方法在各自独立的事务中写入。`startInterview` 和 `nextQuestion` 这些纯数据操作才加 `@Transactional`。」**

### 结构化输出（核心设计）

**Q：详细讲讲你处理 AI 输出不可靠的方案？**

**「我设计了两个层面的容错：**

**第一层 —— 本地 JSON 修复。** 当 AI 返回的内容在 JSON 字符串值内部包含未转义的双引号时（比如参考答案里出现了 `"`），`BeanOutputConverter` 会解析失败。我的修复算法是逐字符扫描 JSON，追踪是否在字符串内部，如果遇到未转义的双引号且不是 JSON 结构边界（即下一个非空字符不是 `,` `}` `]`），就自动加上反斜杠转义。这条路修复成功率大约 60%，能避免很多不必要的 AI 重试调用。

**第二层 —— 带错误上下文的 AI 重试。** 如果本地修复也失败，我会携带上次解析的异常信息重新调用 AI，在提示词里明确告诉它「上次的 JSON 格式有问题，原因是：XXX」。最多重试 2 次，连续失败则抛出明确的业务异常。」

## 工程层（2-3分钟）

### 简历关联出题

**Q：你根据简历出题时，AI 怎么保证题目不超出简历范围？**

**「我在 System Prompt 里做了严格的约束——把简历全文放在提示词中，并明确要求「题目必须围绕简历中提到的技术栈、项目经历和技能来出」「禁止出简历未涉及领域的题目」。**

**同时我让 AI 出的每道题都带 `category` 字段，这样在后续报告生成时，可以按 Category 维度做分数统计，判断候选人在哪些领域强、哪些领域弱。**

**如果用户上传了简历但出题时没关联，或者简历文本为空，系统会退化为纯技能方向出题模式，不再限制简历范围。」**

### 报告缓存

**Q：面试报告生成后，如果用户反复请求，你会重新调用 AI 吗？**

**「不会。报告生成后我把结果存到了 `session.reportJson` 字段，同时把总分和综合评价也冗余存到了 `overallScore` 和 `overallFeedback` 字段。状态更新为 `EVALUATED`。后续请求时检查状态——如果已经是 EVALUATED，直接从数据库反序列化缓存的报告返回，零 AI 调用。」**

### 数据一致性

**Q：如果在出题过程中 AI 返回的题目 JSON 格式正确，但内容少了一道题（比如只返回了 4 道而不是 5 道），你怎么处理？**

**「目前我主要依赖 AI 的 System Prompt 约束来保证输出质量，没有做严格的字段级别校验。如果遇到这种情况，结构化输出框架不会报错——`BeanOutputConverter` 只校验 JSON 结构合法性，不校验语义正确性。**

**如果要加强，我会在 `StructuredOutputService` 层或调用方加一个 Validator 接口，比如「题目数量必须等于 5」「每题必须非空」。验证失败时可以触发重试，在重试提示词里告知 AI 具体哪里不符合要求。」**

## 面试官可能追问

**Q：你说做了重试，那你有没有想过如果 AI 连续重试都返回格式错误，或者 AI 超时了，你的系统怎么表现？**

**A：「我设置了最多重试 2 次，连续失败后抛出 `AI_SERVICE_TIMEOUT` 异常，由全局异常处理器统一返回友好提示。对调用方来说，它会收到一个结构化的错误响应，不会让用户看到 500 白页或者一直转圈。在当前阶段这样够用了。如果生产环境需要更高的鲁棒性，我会考虑加入熔断降级。」**

**Q：为什么出题是 5 道而不是 3 道或 10 道？**

**A：「5 道是我觉得比较合理的数量——太少体现不出候选人的能力分布，太多又会让面试过程冗长。技术上这个值可以做成可配置的，目前写死在 System Prompt 里，未来可以从请求参数动态传入。」**

**Q：你的 System Prompt 里同时塞了简历全文和转换器格式说明，如果简历很长（比如几千字），Token 消耗会不会很大？**

**A：「是的，长简历会导致 Prompt 很长，Token 消耗大。目前的优化方向是简历上传时做 RAG 索引，出题时不是把全文传给 AI，而是先语义检索简历中最相关的片段（比如用户选择了「Java + 高并发」方向，就检索简历中和高并发相关的项目经历），只把这些片段合成出题上下文传给 AI。这个方案已经在简历模块中做了基础设施（pgvector 语义检索），但还没集成到面试出题的链路中。」**

**Q：如果简历文本很长，比如超过 10 万字，你的 JSON 字段存得下吗？**

**A：「我用的是 `columnDefinition = TEXT` 类型，TEXT 在 PostgreSQL 中最大支持 1GB，所以存储层没问题。但我会在应用层限制——简历分块索引时，代码里做了 `MAX_TEXT_LENGTH = 100_000` 的截断。超出部分会被丢弃，对于面试出题场景来说，10 万字的简历已经远远覆盖了核心技术信息。」**
